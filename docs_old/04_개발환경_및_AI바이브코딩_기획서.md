# 04. 개발환경 및 AI 바이브 코딩 기획서

> RuahNote 개발은 ChatGPT Codex를 활용한 AI 바이브 코딩 방식으로 진행한다.  
> 이 문서는 백엔드, DB, 저장소, GitHub, 배포, Firebase 사용 여부, Codex 작업 방식까지 포함한 개발 환경 기준 문서이다.

---

## 1. 문서 목적

이 문서는 RuahNote를 실제로 구현할 때 사용할 개발 환경과 기술 스택을 정의한다.

기능 기획서, 디자인 기획서, DB 기획서는 프로그램의 구조와 기능을 설명하는 문서이고, 본 문서는 다음 내용을 결정한다.

- 어떤 기술로 개발할 것인가
- 백엔드와 DB는 무엇을 사용할 것인가
- GitHub를 어떻게 사용할 것인가
- Firebase를 사용할 것인가
- ChatGPT Codex로 어떤 방식으로 개발할 것인가
- 로컬 개발 환경은 어떻게 구성할 것인가
- 배포는 어디에 할 것인가
- 환경변수와 API 키는 어떻게 관리할 것인가

## 2. 최종 개발 방향

RuahNote는 처음부터 복잡한 네이티브 앱으로 개발하지 않는다.

1차 구현은 **Next.js 기반 웹앱**으로 시작한다.  
이유는 노트, 파일 업로드, 카메라 촬영, OCR, PDF 변환, 구글 캘린더/메일/연락처 연동, AI 요약 기능을 웹 기술과 API 중심으로 구현하기 쉽기 때문이다.

### 개발 원칙

1. ChatGPT Codex로 구현 가능한 단위로 기능을 쪼갠다.
2. 한 번에 전체 앱을 만들지 않고 MVP부터 단계적으로 만든다.
3. 프론트엔드와 백엔드를 하나의 Next.js 프로젝트 안에서 시작한다.
4. DB, 인증, 파일 저장은 Supabase를 중심으로 구성한다.
5. GitHub는 반드시 사용한다.
6. Firebase는 1차 개발에서는 사용하지 않는다.
7. Firebase는 향후 모바일 푸시 알림이나 앱 확장 시 선택적으로 검토한다.
8. 배포는 Vercel을 우선 사용한다.
9. 모든 API 키는 `.env.local`로 관리하고 GitHub에 업로드하지 않는다.
10. Codex 작업 전후로 Git 커밋을 남겨 변경 이력을 관리한다.

## 4. Firebase 사용 여부

### 1차 개발 결론

**Firebase는 1차 개발에서는 사용하지 않는다.**

RuahNote는 단순 채팅앱이나 피드형 앱이 아니라 다음과 같은 관계형 구조가 중요하다.

```text
사용자
└─ 큰 카테고리
   └─ 과목/프로젝트
      └─ 날짜별 노트
         ├─ 첨부파일
         ├─ 녹음파일
         ├─ OCR 결과
         ├─ PDF
         └─ 과제
```

이 구조는 NoSQL 기반으로 설계하는 것보다 PostgreSQL 기반의 관계형 DB로 관리하는 것이 적합하다.  
따라서 DB, 인증, 파일 저장은 Firebase가 아니라 Supabase를 기본으로 사용한다.

### Firebase를 나중에 검토할 수 있는 경우

Firebase는 완전히 제외하는 것이 아니라, 다음 기능이 필요해질 때 선택적으로 검토한다.

- 모바일 앱 푸시 알림
- 웹 푸시 알림 고도화
- Android/iOS 앱 확장
- 사용자 행동 분석
- 모바일 앱 배포 이후 알림 인프라 확장

### Firebase 사용 정책

| 항목 | 1차 개발 | 향후 확장 |
|---|---:|---:|
| Firebase Auth | 사용 안 함 | 필요 낮음 |
| Firestore DB | 사용 안 함 | 사용 안 함 권장 |
| Firebase Storage | 사용 안 함 | Supabase Storage 유지 권장 |
| Firebase Hosting | 사용 안 함 | Vercel 유지 권장 |
| Firebase Cloud Messaging | 사용 안 함 | 푸시 알림 필요 시 검토 |

## 6. 백엔드 사용 계획

### 백엔드 기본 구조

RuahNote의 1차 백엔드는 별도 Express 서버를 만들지 않고 Next.js 내부 서버 기능으로 구현한다.

사용 방식:

- Next.js Route Handler
- Server Actions
- API Routes
- Prisma Client
- Supabase Client

### 백엔드가 담당하는 기능

| 기능 | 처리 내용 |
|---|---|
| 노트 API | 노트 생성, 수정, 삭제, 조회 |
| 카테고리 API | 큰 카테고리 생성, 수정, 삭제 |
| 과목 API | 과목/프로젝트 생성, 정렬, 삭제 |
| 파일 API | 이미지, 녹음, PDF 업로드 |
| OCR API | 이미지 OCR 작업 실행 |
| PDF API | 이미지 기반 PDF, OCR 적용 PDF 생성 |
| 과제 API | 과제 등록, 완료, 보관, 알림 기준 저장 |
| AI API | 요약, 강의노트 생성, 과제 후보 추출 |
| Google API | 캘린더, Gmail, 연락처 연동 |

### 별도 백엔드 서버를 만들지 않는 이유

- 초보자 개발 난이도를 낮출 수 있다.
- Codex가 Next.js 프로젝트 안에서 전체 흐름을 파악하기 쉽다.
- 초기 배포가 단순하다.
- Vercel 배포와 잘 맞는다.
- 나중에 필요하면 AI 작업 큐나 파일 처리 서버만 별도로 분리할 수 있다.

## 7. DB 및 저장소 사용 계획

### DB

**Supabase PostgreSQL**을 사용한다.

주요 데이터:

- 사용자
- 큰 카테고리
- 과목/프로젝트
- 날짜별 노트
- 노트 블록
- 첨부파일
- 녹음파일
- 음성 변환 텍스트
- 강의노트
- 과제
- 스캔 폴더
- 스캔 이미지
- OCR 결과
- PDF 문서
- 구글 계정 연결 정보
- 알림 설정

### ORM

**Prisma**를 사용한다.

Prisma 사용 목적:

- DB 모델을 코드로 관리
- 마이그레이션 관리
- TypeScript 타입 안정성 확보
- Codex가 DB 구조를 이해하기 쉽게 함

### 파일 저장소

**Supabase Storage**를 사용한다.

저장 대상:

- 노트 첨부 이미지
- 카메라 촬영 이미지
- 스캔 이미지
- 녹음파일
- PDF 파일
- 과제 제출 파일
- 기타 첨부파일

파일 자체는 DB에 저장하지 않는다.  
DB에는 파일명, 저장 경로, 파일 크기, MIME 타입, 연결된 노트/과제/스캔 폴더 ID만 저장한다.

## 9. 환경변수 관리

API 키와 비밀값은 `.env.local`에 저장한다.  
`.env.local`은 절대 GitHub에 올리지 않는다.

### .env.local 예시

```env
DATABASE_URL="postgresql://USER:PASSWORD@HOST:PORT/DATABASE"
DIRECT_URL="postgresql://USER:PASSWORD@HOST:PORT/DATABASE"

NEXT_PUBLIC_SUPABASE_URL="https://your-project.supabase.co"
NEXT_PUBLIC_SUPABASE_ANON_KEY="your-anon-key"
SUPABASE_SERVICE_ROLE_KEY="your-service-role-key"

OPENAI_API_KEY="your-openai-api-key"

GOOGLE_CLIENT_ID="your-google-client-id"

GOOGLE_REDIRECT_URI="http://localhost:3000/api/auth/google/callback"
```

## 11. Codex 작업 프롬프트 예시

### 1단계: 기본 레이아웃

```text
Next.js App Router와 Tailwind CSS를 기준으로 RuahNote 기본 레이아웃을 만들어줘.
좌측 사이드바에는 대시보드, 노트, 녹음/강의노트, 과제, 스캔센터, 캘린더, 메일, 연락처, 파일함, 설정 메뉴가 있어야 해.
상단에는 현재 페이지 제목과 빠른 생성 버튼을 배치해줘.
컴포넌트는 components/layout 폴더로 분리해줘.
```

### 2단계: DB 모델

```text
Prisma schema에 User, Workspace, Category, Subject, Note 모델을 추가해줘.
구조는 사용자 > 큰 카테고리 > 과목/프로젝트 > 날짜별 노트가 되도록 관계를 설정해줘.
각 모델에는 createdAt, updatedAt, deletedAt 필드를 포함해줘.
```

### 3단계: 카테고리/과목/노트

```text
큰 카테고리를 생성하고, 그 하위에 과목을 추가하고, 과목 하위에 날짜별 노트를 생성할 수 있는 CRUD API와 화면을 만들어줘.
노트 목록은 날짜 기준 최신순으로 정렬해줘.
```

### 4단계: 이미지 첨부

```text
노트 상세 화면에서 이미지 파일을 업로드하고 Supabase Storage에 저장한 뒤,
해당 이미지 URL을 NoteAttachment 테이블에 저장하는 기능을 만들어줘.
이미지는 노트 본문 아래 첨부파일 영역에 카드 형태로 보여줘.
```

### 5단계: 카메라 촬영

```text
노트 작성 화면에 카메라 촬영 버튼을 추가해줘.
브라우저 MediaDevices getUserMedia API로 카메라 권한을 요청하고,
촬영한 이미지를 미리보기로 보여준 뒤 노트에 첨부할 수 있게 해줘.
이미지는 웹 확인용 중간 품질로 압축해서 저장해줘.
```

### 6단계: OCR

```text
이미지를 첨부하거나 카메라로 촬영한 뒤,
이 이미지에서 텍스트를 추출할까요? 라는 모달을 보여줘.
사용자가 승인하면 OCR 작업을 실행하고,
추출된 텍스트를 미리보기로 보여준 뒤 현재 노트 본문에 삽입할 수 있게 해줘.
```

### 7단계: 스캔센터

```text
스캔센터 메뉴를 만들어줘.
스캔 폴더를 생성하고, 카메라로 여러 장의 이미지를 촬영하면 폴더명_001.jpg, 폴더명_002.jpg 방식으로 넘버링해서 저장해줘.
스캔 완료 후 OCR 변환 또는 PDF 변환으로 이어갈 수 있게 해줘.
```

### 8단계: PDF 변환

```text
스캔 이미지 목록에서 여러 이미지를 선택한 뒤 PDF로 변환하는 기능을 만들어줘.
PDF 변환 전에 OCR 작업을 먼저 진행할 것인지 묻고,
사용자가 승인하면 OCR 후 PDF를 생성해줘.
PDF는 Supabase Storage에 저장하고 DB에 파일 정보를 저장해줘.
```

## 13. 배포 계획

### 1차 배포

**Vercel**을 사용한다.

이유:

- Next.js와 가장 잘 맞는다.
- GitHub 저장소와 쉽게 연동된다.
- 배포 과정이 단순하다.
- 초보자도 운영하기 쉽다.

### 배포 흐름

```text
로컬 개발
→ GitHub push
→ Vercel 자동 빌드
→ 배포 URL 생성
→ 테스트
```

### 배포 전 확인 사항

- `.env.local` 값이 Vercel 환경변수에 등록되어 있는가
- Supabase URL과 Key가 정확한가
- Google OAuth Redirect URI가 배포 주소로 등록되어 있는가
- OpenAI API Key가 등록되어 있는가
- Prisma migration이 Supabase DB에 반영되었는가

## 14. 보안 및 권한 관리

### 기본 원칙

- 사용자는 자신의 데이터만 볼 수 있어야 한다.
- Google access token과 refresh token은 암호화해서 저장해야 한다.
- Supabase Service Role Key는 서버에서만 사용한다.
- 클라이언트에 노출되는 키와 서버 전용 키를 구분한다.
- 파일 접근 권한은 사용자 ID 기준으로 제한한다.
- RLS 정책을 적용한다.

### 민감한 데이터

다음 데이터는 특별히 주의한다.

- 구글 OAuth token
- 이메일 내용
- 연락처 정보
- 녹음파일
- 강의 스크립트
- 과제 파일
- OCR로 추출된 문서 텍스트

## 17. 결론

RuahNote는 ChatGPT Codex를 활용해 단계적으로 구현 가능한 웹앱으로 개발한다.

초기에는 Next.js, Supabase, Prisma, GitHub, Vercel 조합으로 단순하고 안정적인 개발 환경을 만든다.  
Firebase는 1차 개발에 포함하지 않고, 향후 모바일 푸시 알림이나 앱 확장이 필요할 때 선택적으로 검토한다.

이 구조는 초보자가 AI 바이브 코딩으로 기능을 하나씩 구현하기에 적합하며, 나중에 모바일 앱, 데스크톱 앱, 협업 기능으로 확장할 수 있다.
